**预期收益**: -0.15~0.3ms per H2 request（当前 p50 额外开销的主要来源之一）

## P1 — 中等收益，需要一定改动量

### [TASK-D] Http1Driver keep-alive 空闲超时用懒注册 timer

**文件**: `tunnel-lib/src/proxy/http.rs:37`

**问题**:
```rust
tokio::time::timeout(KEEPALIVE_IDLE_TIMEOUT, driver.read_request()).await
```
每个 keep-alive 循环迭代都向 tokio reactor 注册一个 60s 定时器，然后在下一条请求到来时立即取消它。
在 8000 rps 下，每秒创建和销毁 8000 个定时器 handle（heap alloc + wheel insert + wheel remove）。

**修法**（参考 pingora-timeout 思路）:
包装一个 lazy timeout：只有在 `read_request()` 第一次返回 `Poll::Pending` 时才向 reactor 注册 timer。
热路径（请求频繁到达，read 立即 ready）完全跳过 timer 分配。

实现方式：
```rust
struct LazyTimeout<F> { future: F, deadline: Instant, timer: Option<tokio::time::Sleep> }
impl Future for LazyTimeout<F> {
    fn poll(...) {
        if self.future.poll(...).is_ready() { return ready; }
        let timer = self.timer.get_or_insert_with(|| tokio::time::sleep_until(self.deadline));
        if timer.poll(...).is_ready() { return timeout_err; }
        pending
    }
}
```

**预期收益**: 消除 keep-alive 热路径的 timer 分配，8000 rps 下节省 ~8000 次/s heap+wheel 操作

## P2 — 架构级，需要设计确认后再动

### [TASK-G] 分析 egress sender_cache 与 ingress sender_cache 的 H2Sender 生命周期对称性

**背景**:
ingress TLS 路径（`handle_tls_connection`，第 137 行）：每个 TCP 连接建一个 `new_h2_sender()`，连接级别缓存，生命周期与 TCP 连接绑定。
ingress H2 plaintext 路径（`handle_plaintext_h2_connection`，第 207 行）：每个 TCP 连接建一个 `sender_cache HashMap`，按 RouteTarget 分 sender。
egress client entry：每个 QUIC stream 被 server 侧 `ProxyEngine` 接收后，从 `ServerEgressMap` 取 `H2Sender`。

**问题**:
这三条路径的 H2Sender 生命周期不一致：
- ingress TLS：1 TCP → 1 H2Sender（只走一个 proxy_name）
- ingress H2：1 TCP → N H2Sender（按 vhost 分，所以 multihost 压测下 50 VU 同一 TCP 连接看到 50 个 vhost 时只建 50 个 sender，合理）
- egress：每个 QUIC stream 独立，server 侧的 sender 是 per-`(Connection, proxy_name)` 全局缓存（TODO-56 实现后）

**任务**:
画出三条路径各自的 H2Sender 复用边界图，确认 TODO-56 实现后 sender 的 invalidation 路径（Connection 断开 → sender 清理）是否覆盖全部路径，避免内存泄漏。

**文件**: `ci-helpers/warmup.sh`

**问题**:
当前 warmup 是单连接串行探针，只确认"能通"，无法保证：

确保 pool 满连接、sender cache 热、BBR 窗口稳定后再退出 warmup。

**预期收益**: 进一步压缩 bench 早期高延迟样本，降低 p95 1~3ms
